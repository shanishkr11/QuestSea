What is Dependency Injection in ASP.NET Core? DI / IoC
Detailed Answer

Dependency Injection (DI) is a design pattern for achieving loose coupling. Instead of creating dependencies inside a class, they are provided externally. ASP.NET Core has a built-in IoC container that manages the lifetime and resolution of services.

Real-world Example

Register in Program.cs: builder.Services.AddScoped<IUserService, UserService>(). The IUserService dependency is then injected into a controller via constructor injection.

Interview Ready Answer

DI is a technique that helps achieve loose coupling and improves testability and maintainability. ASP.NET Core provides a built-in IoC container to manage service lifetimes: Singleton, Scoped, and Transient.

LifetimeDescription
SingletonOne instance for the entire application lifetime.
ScopedOne instance per HTTP request.
TransientNew instance every time it is requested.
Common Mistakes
  • Using new keyword to create dependencies instead of injecting them.
  • Registering services with the incorrect lifetime (e.g. Singleton holding Scoped).
  • Not disposing scoped services properly.
What is the difference between async/await and Task.Run in C#? C# / Async
Detailed Answer

async/await is used for I/O-bound operations (database calls, HTTP requests) — it frees the thread while waiting without spawning a new one. Task.Run offloads CPU-bound work to a thread pool thread, which is appropriate for heavy computation.

Real-world Example

Use await dbContext.Users.ToListAsync() for database queries. Use Task.Run(() => HeavyComputation()) for image processing or report generation.

Interview Ready Answer

Use async/await for I/O-bound work to avoid blocking threads. Use Task.Run for CPU-bound work to avoid blocking the calling thread. Never use Task.Run in ASP.NET Core controllers for I/O — it wastes thread pool resources.

Common Mistakes
  • Wrapping async I/O calls inside Task.Run unnecessarily.
  • Using .Result or .Wait() which can cause deadlocks.
  • Marking void methods as async instead of returning Task.
Explain Clean Architecture and its layers in .NET. Architecture
Detailed Answer

Clean Architecture separates concerns into concentric layers where inner layers have no dependency on outer ones. The dependency rule states all source code dependencies must point inward.

Real-world Example

Layers in a .NET solution: Domain (entities, value objects) → Application (use cases, CQRS handlers) → Infrastructure (EF Core, repositories) → Presentation (API controllers, minimal APIs).

Interview Ready Answer

Clean Architecture enforces the Dependency Inversion Principle at the architectural level. Business logic in the Application layer is completely isolated from frameworks, databases, and UI — making it independently testable and replaceable.

Common Mistakes
  • Referencing Infrastructure from the Domain layer.
  • Putting business logic directly in API controllers.
  • Skipping interfaces and depending on concrete implementations.
